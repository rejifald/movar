## Permission justifications

Re-used verbatim per store unless a reviewer asks for a longer answer.

- **`storage`** — Persist user preferences and operational state across browser restarts. Preferences (target language, allowlisted sites, hidden languages, popup/options UI language, DOM-modification opt-in) live in `chrome.storage.sync` so they roam with the user's browser profile via Chrome Sync / Firefox Sync, which both encrypt the payload before it leaves the device; Movar runs no server of its own. Per-device operational state — the current pause status and the rolling corrections log (last 1,000 entries, domain only, no URLs or page contents) — lives in `chrome.storage.local` and never leaves the device. See the full breakdown at <https://movar.fyi/privacy>.
- **`declarativeNetRequest`** — Install two dynamic rules driven by the user's language priority, with allowlisted domains excluded from both: (1) rewrite the outgoing `Accept-Language` request header (e.g. `uk,en;q=0.9`) on top-level and sub-frame navigations, so servers return content in the preferred language; (2) redirect `google.*/search` query navigations to the same URL with Google's language parameters (`hl`, `lr=lang_*`) applied and Google's opaque session-bias tokens (`sei`, `gs_lcrp`, …) removed, BEFORE the request leaves the browser — one page load per search instead of a rewrite-after-load (Chrome/Firefox; Safari keeps the in-page fallback rewrite instead). The rules are declarative and touch a single request header plus a handful of URL query parameters — the extension never inspects or modifies request bodies, response bodies, or page content. Sole `declarativeNetRequest` use (`apps/extension/src/lib/dnr.ts`).
- **`alarms`** — Schedule Movar's automatic resume when a timed (1-hour) pause expires (`RESUME_ALARM` in `apps/extension/src/lib/pause.ts`); no other use. This is load-bearing, not convenience: the `Accept-Language` correction is a declarative `declarativeNetRequest` rule (browser-enforced, no per-request JS), so a pause works by switching that rule off — and nothing runs while paused to watch the clock. `alarms` is the only MV3-reliable way to wake the service worker at expiry and switch the rule back on; `setTimeout` dies with the torn-down worker, and a lazy "resync on next popup/navigation" would leave Russian content un-blocked past the hour the user was promised.
- **`<all_urls>` (host permission)** — The content script and the `declarativeNetRequest` rules (Accept-Language everywhere, the pre-request URL rewrite on Google search) apply the language-correction logic on whichever site the user is currently viewing; since the user expects in-language content everywhere on the web, the permission cannot be narrowed to a fixed allowlist. Required at install on Safari; on Chrome/Firefox it's an optional permission requested at runtime instead — a one-click native prompt from the first-run onboarding page (`entrypoints/onboarding`) rather than an unconditional install-time grant. No page content or browsing history ever leaves the device.

**Not a permission, but reviewers will see it — keyboard shortcuts:** the manifest declares a `commands` block (no install warning, not a permission, doesn't change the permissions count). Two commands, dispatched by `commands.onCommand` in [background.ts](apps/extension/src/entrypoints/background.ts): `toggle-pause` (suggested `Alt+Shift+P`) toggles the global pause (timed 1-hour ↔ resume), the same codepath as the popup's pause controls; `reveal-all` (suggested `Alt+Shift+R`) is "show everything on this page" — it forwards `movar:restoreHidden` to the active tab's content script (reusing the popup's reveal handler) and is a no-op on a tab without one. Suggested keys are user-rebindable in `chrome://extensions/shortcuts` (Chrome/Edge) and `about:addons` (Firefox). Safari (#56) lets the user assign them in its own preferences and may ignore `suggested_key`; the `commands` key is valid MV3 on every target, degrading at worst to "no default binding".

**Safari only — `nativeMessaging`:** the Safari build (and only Safari) adds the `nativeMessaging` permission, injected by the `build:manifestGenerated` hook in [wxt.config.ts](apps/extension/wxt.config.ts) — never the manifest literal — so the published Chrome / Firefox / Edge install surface stays the base three permissions (and the README permissions badge, parsed from that literal, is unchanged). It backs the host-app settings bridge: the Movar container app's standalone settings panel writes `MovarSettings` into the `group.fyi.movar.safari` App Group, and the extension reconciles it via `browser.runtime.sendNativeMessage` → `SafariWebExtensionHandler` ([apps/extension/src/lib/native-settings.ts](apps/extension/src/lib/native-settings.ts)). No data leaves the device — it is app↔extension IPC on the same device. This standalone host-app functionality (a local Ukrainian/Russian language checker plus the settings panel) is also Movar's remediation for App Review **Guideline 4.2 (Minimum Functionality)**: the iOS host app must do more than just launch the extension. App Group setup steps: [docs/safari-deploy.md](docs/safari-deploy.md).

**Not a permission, but reviewers will see it:** the manifest declares `web_accessible_resources: [{ resources: ["features/*.js", "models/*.js", "chunks/*.js"], matches: ["<all_urls>"] }]`. These are the off-by-default dynamic capability chunks (the content-hiding feature, the curtain presenter, and the per-site page-content models), split out of `content.js` so their bytes load only when a given page needs them; the content script pulls in the resolved set via `runtime.getURL(...)`, which requires the files to be web-accessible. This grants no new capability and shows no install-time warning — it only exposes the extension's own bundled JS (no user data) to pages the content script already runs on. See `bundleCapabilityChunks` in [wxt.config.ts](apps/extension/wxt.config.ts).

## Per-store accounts & artifacts

All targets ship as **Manifest V3** (`manifestVersion: 3` is forced in [wxt.config.ts](apps/extension/wxt.config.ts) — WXT would otherwise default Firefox to MV2). Firefox floors are pinned to 140 (desktop, July 2025) and 142 (Android, September 2025) — the versions that introduced `browser_specific_settings.gecko.data_collection_permissions`. Older Firefox would silently ignore the declaration, which AMO flags as a `KEY_FIREFOX_*_UNSUPPORTED_BY_MIN_VERSION` warning. The older `declarativeNetRequest` floor (113, May 2023) is now moot.

## Pre-submission verification

Automated checks — run `pnpm verify:release` to do all five in one go:

- [x] `pnpm validate` clean (typecheck + lint + test + publint)
- [x] Chrome and Firefox zips produced under `apps/extension/.output/`
- [x] Zip contents do not include sourcemaps, `.env`, `.DS_Store`, or `node_modules/`
- [x] Firefox zip passes Mozilla's `addons-linter` (the same engine the AMO review pipeline runs) — run via `pnpm verify:release` (step 5/7), which lints the built Firefox zip with `web-ext lint` (tracks `web-ext@latest` to stay aligned with AMO's server-side ruleset). Catches missing `data_collection_permissions`, `strict_min_version` mismatches with declared permissions, and any new AMO policy rule shipped in a future release. Allowlist limited to `UNSAFE_VAR_ASSIGNMENT` from React DOM internals (two `innerHTML` writes inside React's vendored bundle that we cannot remove without dropping React); see `addons_linter_acknowledged` in [scripts/verify-release.sh](scripts/verify-release.sh).

Manual — required after the script is green:

- [ ] Smoke test in Chrome — load `.output/chrome-mv3/` unpacked, exercise popup, change a setting in options, visit a Russian-defaulting Ukrainian site, confirm switch
- [ ] Smoke test in Firefox — same flow against `.output/firefox-mv3/`
- [ ] Smoke test in Safari (macOS) — `pnpm --filter @movar/extension build:safari:app`, `open apps/extension/.output/safari/Movar.app`, enable Movar in Safari ▸ Settings ▸ Extensions (Develop ▸ Allow Unsigned Extensions for the ad-hoc build), then run the same flow. See [docs/safari-deploy.md](docs/safari-deploy.md).
- [ ] **iOS Safari MV3 static-import smoke test** — once Movar ships on iOS, confirm the content script's static import of chrome-ai (from `@movar/lang-detect`) and the background worker's static import of franc (from `@movar/lang-detect/franc`) load without runtime error. Visit a page with no `<html lang>` and a known non-English body, verify the tier-7 path fires per [docs/on-device-language-detection.md](docs/on-device-language-detection.md). Validates the static-import assumption made in the ADR — the riskiest unverified claim for that target.
